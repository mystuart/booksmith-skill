# 全局规则（铁律、禁止事项、已知陷阱）

> Booksmith 的全局规则集。SKILL.md 主流程引用本文件，而非相反。
> 所有写作、排版、命名、调研的铁律，以及禁止事项、已知陷阱和诚实边界，均以此处为准。

---

## 写作铁律

- **逐章顺序写，不可并行**（质量优先于效率）
- **风格锚定不可跳过**（必须先写样板章确认再批量写）
- **上下文累积不可省略**（写第 N 章必须读前 N-1 章）
- **独立验证不可跳过**（Phase 4 必须用子 agent 审查）
- **质量验证迭代上限 2 次**（不无限打磨，标注薄弱项交付）

---

## 排版铁律

- 所有内容元素左右对齐，不允许宽度不一
- 图片不超过正文区域宽度
- 页脚只显示页码，不显示文件路径
- 打印时保留背景色（tip 框、代码块、表头等）
- **版面风格必须显式确认**：排版前确认使用 classic / modern / academic / minimal 中的哪一种
- 技术书默认使用 modern 风格（无衬线正文 + 段落间距分隔）
- 文学/人文类书籍使用 classic 风格（衬线正文 + 首行缩进）
- 代码块使用原生语法高亮，不得转义破坏代码可读性

---

## 命名铁律

- 目录名和文件名一律使用英文（kebab-case），如 `docker-basics/`、`ch01.md`、`01-architecture.md`
- 正文内容使用中文不受此限制
- 书名是中文时，目录名取英文译名或主题关键词的 kebab-case 形式

---

## 调研铁律

- 信源总数 ≥ 20，S/A ≥ 50%
- 关键事实 ≥ 2 个独立信源交叉验证
- 一手源优先：官方文档 > 原始论文 > 权威媒体 > 社区
- 不使用百度百科、百度知道

---

## 禁止事项

### 内容层面

- 不编造信源或引用（有就是有，没有就标注「待验证」）
- 不将同一概念在不同章节完整重复展开（已定义的术语引用即可）
- 不使用「众所周知」「显然」「毋庸置疑」等填充词
- 不在章节末尾写「总结」然后用 bullet 重复正文内容
- 不凭训练数据编造版本号、API 参数、配置项（不确定就标注）

### 交互层面

- 不在生成过程中向用户发过长的中间结果（每章只展示摘要）
- 不跳过用户确认环节（除非用户明确说「直接做」）
- 不在 Phase 3 写作中途停下来问用户「这一段 OK 吗？」（写完整章再展示）

### 排版层面

- 不使用花哨的渐变、阴影、多彩卡片
- 不让章节标题孤行出现在页面底部

## 已知陷阱

- **Typst `rect()` 默认不可跨页**：代码块、提示框、引用块如果内容较长，使用 `rect()` 包裹会导致内容溢出页面（背景只盖第一页，文字漏到页脚）。修复：外层改用 `block(breakable: true)`，内层用 `rect()` 保持样式。
- **`manuscript/` 必须是扁平结构**：PDF 生成脚本按字母顺序读取 `manuscript/*.md`。如果章节放在 `manuscript/phase-1/ch01.md` 等子目录中，脚本会找不到文件。所有 `.md` 必须直接放在 `manuscript/` 下。
- **layout.md 不是 CSS 文件**：排版参数用自然语言描述（如"页面尺寸：A4""行距：1.7em"），各排版引擎按需解析。不要写 CSS 变量语法（`--indigo: #312E81`），除非需要兼容旧项目。
- **长代码块导致标题 orphan**：如果代码块超过一页高度且紧跟在标题后面，Typst 分页算法会把整个不可拆分的块推到下一页，标题被单独留在上一页。使用 `breakable: true` 后可避免。
- **调研 Agent 超时（尤其涉及大型源码分析时）**：Phase 1 并行子 Agent 默认 300s 超时。如果某个 Agent 的任务包含「读取数百 KB 的 C++ 源码 + Web 搜索 + 交叉验证」，极易超时。修复：① 主 Agent 先用 `execute_code` + `grep`/`find` 对本地源码做快速预扫描，提取关键结构和函数名，再让子 Agent 做补充验证；② 超时后主 Agent 直接接管该方向，手动完成调研文件写入；③ 单次 `delegate_task` 不超过 3 个并行子 Agent（`max_concurrent_children` 限制）。
- **`booksmith-typst.py` 的 Python 解释器路径问题**：脚本依赖 `import typst`。系统默认 `python3`（如 macOS `/usr/bin/python3`）可能未安装 `typst` 包，而 conda 环境已安装。修复：排版前先用 `python3 -c "import typst"` 探测，失败时尝试 `$(which python)` 或 conda 路径（如 `/opt/homebrew/Caskroom/miniconda/base/bin/python`）。
- **嵌套 Markdown 代码块的围栏冲突**：当手稿需要展示 Markdown 源码（如 SKILL.md 模板、LLM Prompt 模板）时，外层用 ` ```yaml ` 包裹，内层又用 ` ```markdown `，两者都是 3 个反引号，Markdown 解析器会将内层的 ` ``` ` 误判为外层代码块的结束，导致 `# Role`、`# Task` 等内容泄漏到文章结构中成为章节标题。修复：外层改用波浪号围栏 `~~~yaml ... ~~~`，内层保持反引号 ` ```markdown ... ``` `。同时 `booksmith-typst.py` 的 Markdown→Typst 转换器已更新为同时支持 ` ``` ` 和 `~~~` 两种围栏。排版后可用 `grep "^= " debug.typ` 检查是否出现泄漏的 H1。
- **`anchor-sample.md` 被合并进 PDF**：Phase 2 的风格锚定样板章存放在项目根目录的 `anchor-sample.md`（不在 `manuscript/` 中）。PDF 脚本按字母顺序读取 `manuscript/*.md`，已排除 `glossary.md` 和 `anchor-sample`（以防被误放入 manuscript 目录）。
- **`§01` 章节编号未转为"第一章"**：手稿标题使用 `§01 引言` 格式，`_extract_title` 原来只是去掉前缀，导致 PDF 章节标题没有"第一章"前缀，视觉上与二级标题平级、难以区分。修复：脚本已更新，自动将 `§NN` 转换为"第N章"（1→一，10→十，12→十二，依此类推）。

## 诚实边界

本 Skill 的局限：

- 生成的初稿质量取决于调研深度——调研不足则内容空洞
- 代码示例只做逻辑检查，不保证实际可运行
- 引用 URL 可能在生成后失效
- 不替代专业审校和出版社的三审三校
- 生成的 PDF 排版精度不如专业排版软件（InDesign 等）
- 长文档（30+ 章）的上下文累积可能导致后期章节对前文细节记忆衰减
- 断点续写时，恢复的上下文可能不如连续写作完整
