# 出版风格体系

本文件定义三种出版风格的完整模板。Phase 2 风格锚定时，根据用户选择读取对应章节。

---

## O'Reilly 风格（默认）

对标 O'Reilly Media 的技术书品质。官方 Style Guide 定义的基调：**"conversational, user-friendly, and informational but not chatty"**——像一个有经验的同事在帮新同事上手。

### 单章结构

```
1. 场景引入（1-2 段真实场景或故事，制造「我遇到过这个问题」的共鸣）
   ❌ 禁止：「本章将介绍...」式教科书开头
   ✅ 正确：用一个读者会遇到的真实痛点开场

2. 核心概念讲解（先直觉后精确：类比 → 定义 → 技术细节）
   每个概念配至少 1 个生活化类比

3. 代码/配置示例（如有）
   - 必须可运行，不是片段
   - 带中文注释
   - 每行不超过 75 字符（O'Reilly 标准）
   - 超过 30 行截断并标注 `... (代码省略，完整版见附录)`

4. 「Tip」框 — 最佳实践和快捷方法
   用途：「要快速查找所有修改过的文件，使用 git status -s」

5. 「Warning」框 — 可能导致数据丢失、安全风险的操作
   用途：「注意：此命令会删除所有未提交的更改」

6. 「Note」框 — 有用但非关键的信息
   用途：「此函数在 3.0 版本中已弃用」

7. 章节小结（3-5 个要点，提炼洞察而非复述正文）

8. 延伸阅读（2-5 条，附 URL，引用调研信源）
```

### 四字体规范（O'Reilly 最独特的约定）

这是 O'Reilly 书籍辨识度最高的排版特征，必须在 HTML 中严格遵循：

| 语义 | HTML 样式 | 用途 | 示例 |
|------|---------|------|------|
| 新术语 / 强调 / 文件名 | `<em>` 斜体 | 首次出现的术语、文件路径、书名 | `README.md`、*延迟计算* |
| 代码元素 | `<code>` 等宽 | 函数名、变量名、类名、HTML 标签、命令名 | `print()`、`myVariable`、`<div>` |
| 可替换项 | `<code><em>` 等宽斜体 | 读者需要替换的占位内容 | `python <em>scriptname</em>.py` |
| 用户输入 | `<code><strong>` 等宽加粗 | 读者应该输入的内容 | 代码块中的用户输入部分 |

### 标题大小写规则

O'Reilly 对标题大小写有严格规定：

- **A 级 / B 级标题**：Title Case（每个主要单词首字母大写）
  - 例：「Understanding the Component Lifecycle」
- **C 级及以下标题**：Sentence Case（仅首字母和专有名词大写）
  - 例：「How to handle edge cases」
- 中文标题不适用此规则

### 其他 O'Reilly 约定

- **序列逗号**（Oxford comma）强制使用：「红、白、和蓝」
- **图表编号按章节**：Figure 1-1、Table 3-2、Example 5-3
- **代码示例编号**：Example 3-1（第 3 章第 1 个示例）
- **交叉引用格式**：「如第 3 章所述」「见图 2-1」

### 写作基调

- **人称**：第二人称「你」或第一人称复数「我们」，营造同行交流感
- **节奏**：概念 → 类比 → 示例 → 踩坑，循环推进
- **类比偏好**：生活化场景（「想象你是一个邮递员...」），避免过于学术
- **确定性**：有定论用肯定语气，有争议标注「社区对此有不同看法」
- **无填充词**：不用「众所周知」「显然」「毋庸置疑」

### 单章字数目标：3000-5000 字，代码占比 15-25%

### HTML 标记

```html
<div class="chapter">
  <h2>Chapter XX — [Title Case 标题]</h2>
  <div class="ch-intro">[场景引入]</div>

  <!-- A 级标题：Title Case -->
  <h3>[A 级标题]</h3>

  <!-- 核心概念 -->
  <div class="term">
    <div class="term-name">中文术语</div>
    <div class="term-en"><code>EnglishTerm</code></div>
    <p>解释正文...</p>
  </div>

  <!-- Tip 框 -->
  <div class="tip tip-tip">
    <strong>提示：</strong>[最佳实践...]
  </div>

  <!-- Warning 框 -->
  <div class="tip tip-warning">
    <strong>警告：</strong>[可能导致问题的操作...]
  </div>

  <!-- Note 框 -->
  <div class="tip tip-note">
    <strong>注意：</strong>[补充信息...]
  </div>

  <!-- 代码示例 -->
  <pre class="codeblock"><code># Example XX-1: [描述]
<em>可替换部分</em>  <!-- 等宽斜体 -->
<strong>用户输入部分</strong>  <!-- 等宽加粗 -->
</code></pre>

  <!-- 小结 -->
  <div class="summary">
    <h3>本章要点</h3>
    <ul><li>...</li></ul>
  </div>

  <!-- 延伸阅读 -->
  <div class="further-reading">
    <h3>延伸阅读</h3>
    <ul><li><a href="URL">标题</a> — 一句话说明</li></ul>
  </div>
</div>
```

## Academic 风格

对标 Springer LNCS / 学术专著——严谨客观，数据驱动，形式化结构。

### 单章结构

```
1. 章节引言（1 段，说明本章讨论什么、与前章的关系）
   无需「摘要」标签（monograph 风格），直接用引言段落

2. 编号章节（1.1, 1.2, 1.3...）
   每节聚焦一个子主题，有明确论点

3. 子节（1.1.1, 1.1.2...）
   深入细节，C 级标题用 Sentence Case

4. 案例分析（基于调研数据，引用 ≥3 个来源）

5. 章节结论（总结论点 + 局限性 + 未来方向）

6. 参考文献（格式严格的编号列表）
```

### 引用规范

- **内联标注**：`[3]` 或 `[3, 7]` 或 `[3-5]`
- **参考文献格式**：
  ```
  [1] 作者, 书名, 第X版, 出版社, 出版地 (年份).
  [2] 作者, 文章标题, 期刊名 42(3), 123-145 (2023).
  [3] 作者, 论文标题, in: 会议名, pp. 1-10, 出版社 (2023).
  ```
- **et al.**：正文引用 3+ 作者时用 `[Smith et al.]`，参考文献列表列出全部作者
- 一手源（论文、官方文档）占比 ≥ 60%

### 图表规范

- **编号按章节**：Fig. 1.1、Table 3.2
- **图标题**在图下方，**表标题**在表上方
- 所有图表必须在正文中被引用
- 定理/定义/引理各自独立编号：Definition 3.1、Theorem 3.2

### 写作基调

- **人称**：第三人称为主，「研究表明...」「本章提出...」
- **第一人称复数**「我们」可用于描述作者贡献：「我们提出了一种新方法...」
- **被动语态**在方法章节中常见：「该算法在三个数据集上进行了评估...」
- **避免**：口语化表达、缩写、非正式用语
- **确定性**：用「数据表明」「实验证实」，避免「显然」「毫无疑问」
- **中立性**：有争议的话题呈现多方观点并标注「尚无定论」

### 代码密度

- 低（5-15%），侧重概念和架构
- 代码仅用于关键算法或协议说明
- 用伪代码或架构图描述替代长代码

### 单章字数目标：4000-6000 字，代码占比 5-15%

### HTML 标记

```html
<div class="chapter">
  <h2>Chapter XX — [标题]</h2>
  <p>[引言段落]</p>

  <h3>X.1 [子节标题]</h3>
  <p>正文...引用格式 [3]。</p>

  <h4>X.1.1 [子子节标题，Sentence Case]</h4>
  <p>...</p>

  <!-- 图：标题在下方 -->
  <figure>
    <img src="fig-X-1.png" alt="描述">
    <figcaption>Figure X.1: [图标题]</figcaption>
  </figure>

  <!-- 表：标题在上方 -->
  <table>
    <caption>Table X.1: [表标题]</caption>
    <thead><tr><th>...</th></tr></thead>
    <tbody><tr><td>...</td></tr></tbody>
  </table>

  <!-- 定义/定理 -->
  <div class="definition">
    <strong>Definition X.1</strong> (术语名): [定义内容]
  </div>

  <!-- 结论 -->
  <div class="chapter-conclusion">
    <h3>结论</h3>
    <p>...</p>
  </div>

  <!-- 参考文献 -->
  <div class="references">
    <h3>参考文献</h3>
    <ol>
      <li>作者, 标题, 来源, 年份. <a href="URL">链接</a></li>
    </ol>
  </div>
</div>
```

## Handbook 风格

对标 Microsoft Writing Style Guide / Google Developer Docs——任务导向，步骤明确，可直接执行。

### 单章结构

```
1. 问题陈述（一句话说清楚要解决什么）

2. 前提条件（Prerequisites）
   - 用 bullet 列表列出开始前必须具备的条件
   - 复杂前提链接到其他章节，不在此展开

3. 解决方案概览（整体思路，1-2 句话）

4. 分步操作（核心部分，见下方步骤规则）

5. 检查清单（用 checkbox 列表）

6. 故障排除 / FAQ（3-5 个，问题-症状-原因-解决方案格式）

7. 下一步（指向下一章或进阶资源）
```

### 步骤写作规则（来自 Microsoft Style Guide）

1. **一步一操作** — 不要在单个步骤中合并多个操作
2. **祈使语气** — 「点击 **保存**」而非「你应该点击保存」
3. **每步不超过 7 条** — 超过就拆分为子组，加子标题
4. **完整句子** — 每步以句号结尾
5. **一致结构** — 所有步骤以动词开头
6. **单步操作用 bullet** — 只有 1 步时不用编号列表
7. **关键步骤后注明结果** — 「此时会弹出确认对话框」
8. **UI 导航路径用 >** — 「选择 **文件** > **另存为**」
9. **交互元素加粗** — 按钮、菜单项、标签页用 `<strong>` 包裹
10. **代码必须可复制粘贴** — 提供完整可用代码，不是片段

### 写作基调

- **人称**：第二人称「你」（直接对话）
- **节奏**：问题 → 前提 → 步骤 → 验证，每步可独立执行
- **时态**：用现在时：「系统显示确认信息」而非「系统将会显示」
- **禁止**：大段理论铺垫，每段必须伴随可操作内容
- **禁止**：模糊指令（「配置好设置」→ 应写「在设置面板中勾选 启用调试模式」）

### 故障排除格式

```
### 问题：[用户视角的症状描述]

**症状**：[用户看到什么]
**原因**：[为什么会发生]
**解决方案**：
1. [步骤1]
2. [步骤2]
```

### 代码密度

- 高（30-50%），手把手级
- 每个步骤都配完整可运行的代码/命令
- 命令用 `<span class="code">` 包裹
- 输出示例用 `<pre class="codeblock">` 包裹
- 不同操作系统用 tab 分隔：Linux | macOS | Windows

### 单章字数目标：2000-4000 字，代码占比 30-50%

### HTML 标记

```html
<div class="chapter">
  <h2>Chapter XX — [标题]</h2>
  <div class="problem">[问题陈述]</div>

  <!-- 前提条件 -->
  <div class="prerequisites">
    <h3>前提条件</h3>
    <ul>
      <li>Node.js 18 或更高版本</li>
      <li>已完成第 X 章的配置</li>
    </ul>
  </div>

  <!-- 分步操作 -->
  <div class="step">
    <h3>步骤 1：[动词 + 名词]</h3>
    <p>[操作描述]。</p>
    <p>此时会看到 [预期结果]。</p>
  </div>

  <div class="step">
    <h3>步骤 2：[动词 + 名词]</h3>
    <p>选择 <strong>文件</strong> > <strong>另存为</strong>。</p>
    <pre class="codeblock"><code>npm install --save <em>package-name</em></code></pre>
  </div>

  <!-- 检查清单 -->
  <div class="checklist">
    <h3>检查清单</h3>
    <ul>
      <li>☐ [验证项1]</li>
      <li>☐ [验证项2]</li>
    </ul>
  </div>

  <!-- 故障排除 -->
  <div class="troubleshooting">
    <h3>故障排除</h3>
    <div class="issue">
      <h4>问题：[症状]</h4>
      <p><strong>原因：</strong>[解释]</p>
      <p><strong>解决方案：</strong></p>
      <ol><li>[步骤]</li></ol>
    </div>
  </div>
</div>
```

## 自定义风格

用户指定自定义风格时，按以下维度记录并生成模板：

| 维度 | 需确认内容 |
|------|----------|
| 单章结构 | 分哪几个板块？顺序是什么？ |
| 写作基调 | 对话式/学术/指令式？ |
| 类比偏好 | 用不用类比？偏好哪类场景？ |
| 代码密度 | 高/中/低？占比多少？ |
| 引用密度 | 每个论点需要几个引用？ |
| 字数目标 | 每章多少字？ |
| 特殊排版 | 有没有独特的排版约定（如 O'Reilly 的四字体）？ |
| 禁止事项 | 有没有不想出现的元素？ |

确认后，按相同格式生成 HTML 标记模板，存入 `references/style-guide.md` 追加 section。

## 风格锚定执行规范

无论哪种风格，锚定流程相同：

1. **选模板**：根据用户选择，读取上方对应风格的完整模板
2. **写样板**：用第一章内容，严格按模板结构写完整一章
3. **用户确认**：展示样板章，确认：
   - 整体节奏和深度是否满意？
   - 各板块是否都需要？
   - 有没有想增减的板块？
   - 特殊排版约定（如四字体、编号格式）是否正确？
4. **锁定**：确认后存入 `anchor-sample.md`，后续所有章节以此为基准
5. **微调**：如果用户说「X 板块去掉」「Y 板块加长」，更新模板和样板

**锚定样板是全书风格一致性的唯一判据。** 写作过程中如果发现风格漂移，以样板为标准纠偏。

## 禁用词表

以下词汇在正文中禁止出现，必须用更具体的表达替代：

| 禁用 | 替代 |
|------|------|
| "众所周知" | 删除，或用具体事实替代 |
| "显然"、"毫无疑问" | 直接陈述事实 |
| "值得注意的是" | 直接说 |
| "综上所述"、"总的来看" | 直接写结论段落 |
| "接下来我们将..." | "我们来..." |
| "进行操作"、"实现功能" | "点击"/"输入"/"做到"/"搞定" |
| "首先...其次...最后" | "第一"/"再说"/"还有一件事" |
| "强大的"、"革命性的"、"颠覆性的" | 用具体数据和事实替代 |

## 禁用开头清单

以下模式出现在章节开头 → 必须重写该章节：

```
❌ "在当今这个AI时代..."
❌ "随着科技的飞速发展..."
❌ "近年来，XX受到了广泛关注..."
❌ "众所周知..."
❌ "今天我们来聊聊XX..."
❌ "XX是什么？让我们一起来看看吧。"
```

好的开头标准：前 2-3 段内必须有具体信息（时间/数字/具体场景/真实痛点）。
